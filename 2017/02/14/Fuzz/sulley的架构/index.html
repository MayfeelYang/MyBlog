sulley架构图

四个主要组成部分

  • Data Generation
    • You build requests out of primitives and legos
    • Legos are complex types that extend the framework
  • Session Management / Driver
    • Requests are chained together in a graph to form a session
    • The session class exposes a standalone web interface for monitoring and control
    • The driver ties targets, agents and requests together
  • Agents
    • Interface with the target for instrumentation and logging purposes
  • Utilities
    • Standalone command line utilities that perform a variety of task

常规用法

  1. Sniff some traffic or reverse some protocol parsing binary
  2. Break the target protocol into individual requests
  3. Represent each request with a series of primitives
    • Individual requests can be tasked out to different people
  4. Setup some targets in conjunction with various agents
  5. Write a driver script which instantiates a session and ties requests, agents and the targets together
  6. Fuzz!
  7. Review results

Session

这个例展示了多种路径和深度。

Agents

  • A flexible sub-system allows you to create custom agents
  • Client-server communication is extremely simple over ”PedRPC”
    • Create a class that extend from pedrpc.server
    • Instantiate pedrpc.client
    • Call class members as if they are local
  • Some agents have already been developed…

Driver

  • The driver is where it all comes together:
    • Import requests from the request library
    • Instantiate a session instance
    • Instantiate and add target instances to the session
    • Interconnect the requests to form a graph
    • Start fuzzing
  • This is where edge and pre/post send callbacks should be defined
  • The driver is entirely free form, though must of them will follow a simple and similar structure

Driver的例子

如上所示的driver的例子就可以展示一个大概的sulley的工作流程：

  1. 初始化一个session
  2. 初始化一个target
  3. 为target设置监控（netmon / procmon / vmcontrol）
  4. 启动target
  5. 将target添加至session
  6. 用户可自定义pre_send函数和post_send函数
  7. session.connect
  8. 进行模糊测试（session.fuzz）

sulley的监控

Agents组件来实现，现有的agent包括：

  • network_monitor
  • process_monitor
  • vmcontrol

监控部分的脚本都引入了sulley中的pedrpc，客户端和服务器的通信就建立在其上，具体内容与实现如下图所示。

vmcontrol

  • Exposes network API for VMWare instrumentation
    • Simple PedRPC wrapper around vmrun.exe
    • Start, stop, suspend, snapshot, revert, etc…
  • Used to restore target health after a fault is induced

vmcontrol.py中实现了两个类：

  • vmcontrol_pedrpc_server: 继承了pedrpc.server
  • vboxcontrol_pedrpc_server: 继承了vmcontrol_pedrpc_server

vmcontrol_pedrpc_server

为了控制VMWare的虚拟机。
实例化该类是必须指定：

  • hostname/IP（host）
  • port（port）
  • vmrun.exe的路径（vmrun）
  • vmx文件的路径（vmx）

该类中有一个较为重要的函数 vmcommand，此函数用于执行指定的虚拟机命令，使用了os.popen(command); out = pipe.realines()进行实现，通过out的值来判断是否成功执行，如果执行失败将不断的再次尝试执行，直至成功。可以执行的命令有：

  • delete_snapshot
  • list 列出正在运行的镜像
  • list_snapshots
  • reset
  • revert_to_snapshot
  • snapshot
  • start
  • stop
  • suspend

通过对以上命令的调用，实现了一下函数：

  • is_target_running
    1. list
    2. 判断list中是否有self.vmx，若有则返回True；否则，返回False
  • wait
    1. 无限循环，此循环的终止条件为：is_target_running()返回True
  • restart_target
    1. revert_to_snapshot
    2. start
    3. wait

vboxcontol_pedrpc_server

为了控制Oracle的VirtualBox虚拟机。
实例化该类是必须指定：

  • hostname/IP（host）
  • port（port）
  • VBoxManage的路径（vmrun）
  • 需要控制的VirtualBox虚拟机的名字（vmx）

可以执行的命令较VMWare多了以下几个：

  • pause
  • resume

同样，还实现了下列函数：

  • get_vminfo
  • is_target_running
  • is_target_paused

process_monitor

  • PyDbg based fault monitoring engine
  • Used to monitor target health
  • Detected faults are catalogued in a “crash bin”
    • Allows for simplistic (backtrace driven) fault clustering
    • More on this later

process_monitor中实现了两个类：

  • DebuggerThread
  • ProcessMonitorPedrpcServer

DebuggerThread

继承了threading.Thread类。
实例化是必须指定的参数是：

  • process_monitor => self.process_monitor
  • process => self.process_name

实现了以下几个函数：

  • init: Instantiate a new PyDbg instance and register user and access violation callbacks. 使用了pydbg.pydbg()初始化了pydbg的实例。
  • dbg_callback_access_violation: Ignore first chance exceptions. Record all unhandled exceptions to the process monitor crash bin and kill the target process. 当监控目标出现异常时调用此回调函数。
  • dbg_callback_user: The user callback is run roughly every 100 milliseconds (WaitForDebugEvent() timeout from pydbg_core.py). Simply check if the active flag was lowered and if so detach from the target process. The thread should then exit. 监控用户的指令，如果用户要求停止监控，则detach监控进程。
  • run: Main thread routine, called on thread.start(). Thread exits when this routine returns.
    self.watch(); self.dbg.attach(self.pid); self.dbg.run()
  • watch: Continuously loop, watching for the target process. This routine “blocks” until the target process is found. Update self.pid when found and return. 如果当前的self.pid为空，则调用pydbg的enumerate_processes函数，通过比对当前的self.proc_name来找到进程号。

ProcessMonitorPedrpcServer

继承了pedrpc.server。
实例化必须指定的参数有：

  • hostname/IP
  • port
  • crash_filename: Name of file to (un)serialize crash bin to/from
  • 还有个proc参数，默认值为None，但是这个参数应该是必须指定的，否则应该无法正常工作吧~

实现的几个重要的函数如下：

  • pre_send(): This routine is called before the fuzzer transmits a test case and ensure the debugger thread is operational.
    • 参数：test_number(Integer)【Test number to retrieve PCAP for.】
    • self.crash_bin.import_file(self.crash_filename)

  • post_send(): This routine is called after the fuzzer transmits a test case and returns the status of the target.

    • 判断当前dbg是否处于access violation，若是，则等待其完成
    • self.crash_bin.export_file(self.crash_filename)

  • start_target: Start up the target process by issuing the commands in self.start_commands.

    • subprocess.Popen(command)，command是self.start_commands中的。
  • stop_target: Kill the current debugger thread and stop the target process by issuing the commands in self.stop_commands.
    • 使用了os.system(command)
